## The REAL install interface (what emit.ts must target)

Source of truth: `packages/accounts/src/smart_account/mod.rs` (trait `SmartAccount`),
`packages/accounts/src/smart_account/storage.rs` (types), and
`packages/accounts/src/policies/spending_limit.rs` (install constraints).

### Two install paths exist

1. **`add_context_rule`** (new rule + policy install in one call) — what this proof used:

```rust
fn add_context_rule(
    e: &Env,
    context_type: ContextRuleType,   // MUST be CallContract(token_address) for spending-limit
    name: String,                    // <= 20 bytes (MAX_NAME_SIZE)
    valid_until: Option<u32>,        // ledger seq, None = no expiry
    signers: Vec<Signer>,            // Signer::External(verifier, pubkey_bytes) | Signer::Delegated(addr)
    policies: Map<Address, Val>,     // policy_contract -> install params Val
) -> ContextRule                     // returned ContextRule.id is the rule id needed for auth later
```

2. **`add_policy`** (attach to an EXISTING rule — the rule must already be
   `CallContract(token)` type):

```rust
fn add_policy(
    e: &Env,
    context_rule_id: u32,            // REQUIRED, missing from current emit.ts sketch
    policy: Address,
    install_param: Val,              // SpendingLimitAccountParams as ScVal map
) -> u32                             // policy id
```

Both require auth from the smart account itself
(`e.current_contract_address().require_auth()`), i.e. a full `__check_auth`
round with an `AuthPayload` signature — a plain
`stellar contract invoke --source-account deployer` CANNOT authorize them
unless the client builds the custom `AuthPayload` ScVal (see below). The
constructor path (`--policies '{...}'` at deploy time) does NOT work for
spending-limit because the constructor creates a `Default`-type rule and
`spending_limit::install` panics with `OnlyCallContractAllowed` (#3227) for
anything but `ContextRuleType::CallContract`.

### Install param encoding (SpendingLimitAccountParams as Val)

ScVal map, keys sorted alphabetically:

```
ScVal::Map([
  { Symbol("period_ledgers") : U32(17280) },
  { Symbol("spending_limit") : I128(6000000) },
])
```

stellar-cli JSON for the same Val: '{"period_ledgers":17280,"spending_limit":"6000000"}'
(i128 as string). The CLI sketch that matches the real interface for a fresh setup:

```bash
stellar contract invoke --network testnet --source-account <feepayer> \
  --id "$SMART_ACCOUNT" \
  -- add_context_rule \
  --context_type '{"CallContract":"'$TOKEN'"}' \
  --name '"spend_limit"' \
  --valid_until null \
  --signers '[{"External":["'$VERIFIER'","'$SIGNER_PUBKEY_HEX'"]}]' \
  --policies '{"'$POLICY'":{"period_ledgers":17280,"spending_limit":"6000000"}}'
```

with the caveat that the smart-account auth entry still needs the custom
`AuthPayload` signature, so the CLI alone can only produce the unsigned XDR
(`--build-only`); signing requires a custom client (or brozorec/smart-account-sign).

### Authorization / invoke interface (`__check_auth`)

`Signature` type of the account contract:

```rust
struct AuthPayload {
    signers: Map<Signer, Bytes>,     // signer -> raw signature bytes (64B ed25519)
    context_rule_ids: Vec<u32>,      // one rule id per auth context, index-aligned
}
```

ScVal encoding (map keys sorted): `{ Symbol("context_rule_ids"): Vec<U32>,
Symbol("signers"): Map<Signer, Bytes> }` where
`Signer::External(verifier, key)` is `Vec[Symbol("External"), Address, Bytes]`.

Signers do NOT sign the host `signature_payload` directly:

```
auth_digest = sha256(signature_payload || xdr(ScVal::Vec[U32(rule_id), ...]))
```

`signature_payload` is sha256 of `HashIdPreimage::SorobanAuthorization{network_id,
nonce, signature_expiration_ledger, root_invocation}`. The ed25519 verifier
example verifies the 64-byte signature over the raw 32-byte `auth_digest`.

A transfer through the smart account is a direct SAC invocation
`transfer(smart_account, to, amount)` whose smart-account auth entry carries
`AuthPayload{signers:{External(...): sig}, context_rule_ids:[1]}` selecting the
CallContract rule; `do_check_auth` then calls `policy.enforce(context,
matched_signers, rule, smart_account)` and the policy intercepts
`fn_name == "transfer"`, reading `args[2]` as the i128 amount.

Working reference client: `/tmp/oz-soroban/client/sa.js` (Node,
@stellar/stellar-sdk 15.1.0). Two-pass simulation is required: recording sim
discovers the auth entry/nonce (it skips `__check_auth`, so its footprint is
incomplete), then an enforcing sim with the signed entry produces the real
footprint and fees.

## Divergences between docs/emit.ts and reality

1. **emit.ts `installSketch` is wrong on the function shape**: it emits
   `add_policy --policy ... --install-param ...` with no `context_rule_id`.
   `add_policy(context_rule_id, policy, install_param)` requires the rule id,
   and the CLI flag is `--install_param` (underscore), not `--install-param`.
   For a from-scratch install the right call is `add_context_rule` (see above),
   since `add_policy` presupposes an existing `CallContract(token)` rule.
2. **`--source-account deployer` cannot authorize the install**: both
   `add_policy` and `add_context_rule` require the smart account's own auth
   (`__check_auth` with `AuthPayload`). The sketch's implication that a normal
   deployer-signed invoke installs the policy is false; a custom signing client
   is mandatory (or the rule must be created in the constructor, which the
   spending-limit policy forbids for `Default` rules: error #3227
   `OnlyCallContractAllowed`).
3. **Spending-limit is token-scoped**: install only succeeds on a
   `ContextRuleType::CallContract(token)` rule, pinning the limit to one token
   contract (here the XLM SAC). `SpendingLimitAccountParams` alone is not
   install-ready; the token address is a required third input the tool should
   surface.
4. **Plain `cargo build --target wasm32v1-none` fails**: the workspace enables
   soroban-sdk feature `experimental_spec_shaking_v2`, which hard-requires
   `stellar contract build` (CLI >= 25.2.0). The README's `stellar contract build`
   is correct; generic cargo instructions elsewhere are not.
5. **README is otherwise accurate** for deploy flow and the auth-digest
   formula (`sha256(signature_payload || context_rule_ids.to_xdr())`), which was
   verified by successful on-chain auth here.
6. **Boundary semantics**: the limit is inclusive (`total + amount > limit`
   blocks), proven by tx 4 (exact 6,000,000 passes) and tx 5 (+1 stroop fails).
7. **Failed-tx capture detail**: an over-cap transfer normally dies in
   enforcing simulation (wallets never submit it). To land an on-chain FAILED
   tx as proof, the client reuses the footprint/resources from an
   under-cap "donor" simulation with the same nonce, then submits the over-cap
   variant (sa.js donor path).
